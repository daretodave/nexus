# plan

The build plan for <PROJECT>. Read by every shipping skill
(`/ship-a-phase`, `/ship-data`, `/iterate`, `/march`,
`/plan-a-phase`, `/critique`, `/triage`, `/oversight`) to keep
the autonomous loop coherent.

## How the loop reads this folder

1. **Status check** тАФ read the "Status (at-a-glance)" block at
   the top of `steps/01_build_plan.md`. The first `[ ]` row is
   the next phase.
2. **Brief** тАФ load `phases/phase_<N>_<topic>.md`. If absent,
   `/ship-a-phase` calls `/plan-a-phase` inline to generate
   one + commit separately.
3. **Bearings** тАФ `bearings.md` holds stack pins, contracts,
   visual defaults, sub-agent registry, standing decisions.
   Read once per session.

## Where design lands

The user emits design exports asynchronously into `design/` at
the project root. The loop does not wait for design тАФ if the
export isn't there, the next phase that touches that surface
integrates it when it lands.

## How the audit works

`/iterate` writes findings to `plan/AUDIT.md` with this format:

```markdown
# Site audit тАФ <ISO date>

## Top 5 findings (scored)

### [8.1] <one-line description>
- category: <content-gaps | data-gaps | seo | links | a11y | tests | perf | external-critique | external-issue>
- impact: <0-10>
- ease: <0-10>
- next: <action тАФ invocation, sub-agent, or follow-up>
```

Entries are flipped `[ ]` тЖТ `[x]` (with commit hash) as
addressed.

A line `> Bias: <category> (set via oversight <date>)` at the
top of `AUDIT.md` weights that category 1.5x in `/iterate`'s
scoring. Set / cleared via `/oversight reset`.

## Manually ticking phases

If a phase ships outside the loop (rare), update
`steps/01_build_plan.md` by hand: flip `[ ]` тЖТ `[x]`, append
the commit hash, add a line to "Phase log". The loop relies on
these markers.

## Related skills

- `skills/ship-a-phase.md` тАФ ship one phase end-to-end.
- `skills/ship-data.md` тАФ ship one data record (if applicable).
- `skills/plan-a-phase.md` тАФ refine a phase brief without
  shipping code.
- `skills/iterate.md` тАФ audit + ship one improvement; closes
  GitHub issues when their fix lands.
- `skills/critique.md` тАФ external-observer pass; writes to
  `CRITIQUE.md`.
- `skills/triage.md` тАФ issue review; routes open issues into
  `AUDIT.md` / `BACKLOG.md` / build plan.
- `skills/expand.md` тАФ plan-expansion pass; reads signals,
  proposes phase candidates to `PHASE_CANDIDATES.md`.
  Posture-controlled (bold default; strict opt-out).
- `skills/march.md` тАФ outer dispatcher (used with `/loop`).
- `skills/oversight.md` тАФ user-in-the-loop adjustment (the only
  skill that asks questions). Promotes phase candidates.
